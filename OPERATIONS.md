# KTTA Platform 運用マニュアル

釧路卓球協会 (Kushiro Table Tennis Association) 様向けの本番運用ガイド。

---

## 1. システム全体像

```
[参加団体]                                  [協会・運営]
    │                                            │
    │ 申込 (Jimdo埋込フォーム)                       │
    ▼                                            │
[Google Apps Script]                              │
    │                                            │
    ▼                                            │
[Google スプレッドシート]──── 集計・領収書発行 ──→  │
                                                 │
                                                 ▼
                                       [KTTA Platform サーバー]
                                              (Node.js + SQLite)
                                                 │
                                                 ├── 管理画面 /admin
                                                 ├── 結果閲覧 /viewer
                                                 └── 観戦ビュー /viewer/live/
                                                       │
                                                       ▼
                                                  [参加者・保護者]
                                                  スマホでアクセス
```

### 2系統運用 (役割分担)
- **Google スプレッドシート系**: 申込受付・集計・領収書 (Google アカウントのみで運用可)
- **KTTA Platform 系**: 大会当日の進行管理・結果記録・観戦ビュー (要 VPS サーバー)

両系統を独立に運用可。完全 Google 系のみでの運用も可能 (進行管理のみ手動なら)。

## 2.5 会場オフライン運用（WiFiが不安定/1時間毎に落ちる会場・強く推奨）

会場のWiFiが**1時間毎に切れる**ような環境では、クラウドサーバに依存した運用は**断のたびに2〜3台すべてが止まり**ます。
**本部1台でローカルサーバを動かし、他端末は会場内ローカルネットワークで接続する**ことで、**会場WiFi(インターネット)が
落ちても運用が一切止まらない**構成にできます。

### 構成図
```
        [本部PC] ── KTTA をローカル起動(standalone) = 大会の"正本"
           │  (WiFiテザリング or モバイルルータ で 会場内ローカル網)
   ┌───────┼───────┬───────────┐
 [運営2台目] [大画面] [観客スマホ]  …全部ローカル網で本部に接続
           ╎
           ╎ インターネットが生きた間欠だけ
           ▼
     [クラウド公開ビュー] ← 観客/不在者が数分遅れで閲覧(任意)
```
ポイント: **ローカル網(本部のテザリング/モバイルルータ)は会場WiFiと別物**なので、会場のインターネットが落ちても
ローカル網は生き続け、本部⇄運営端末は途切れません。

### 当日の手順
1. **本部PCで起動**: `standalone/start.command`(Mac) / `start.bat`(Win) をダブルクリック。コンソールに
   `本部運営: http://192.168.x.x:3000/admin` のように**LAN URLが表示**されます。
2. **ローカル網を作る**: 本部PCの**WiFiテザリング(ホットスポット)**をON、または**モバイルルータ**を置き、
   運営端末・大画面・観客をそこに繋ぐ(会場WiFiには繋がなくてよい)。
3. **他端末を接続**: 本部の管理画面 → ⚙設定 → **「📱端末接続(会場LAN)」** を開き、表示される**URL/QR**を
   他端末で開く(QRはオフラインでもローカル生成)。運営端末は管理キーを入力。
4. **運用**: 抽選・結果入力・台割を本部(正本)で実施。LANが一瞬切れても**操作は各端末に保存され**、戻ると自動送信
   (画面上部の通信バーに「未送信N件」「今すぐ送信」)。**リロードしても画面は出ます**(Service Workerキャッシュ)。
5. **クラウド公開(任意)**: 本部PCの環境変数 `SYNC_CLOUD_URL`(クラウドのURL) と `SYNC_KEY`(クラウドと同値)を
   設定しておくと、**インターネットが生きた間欠で進行中大会をクラウド公開ビューへ自動同期**(観客は数分遅れで閲覧)。
   手動同期は「📱端末接続/クラウド同期」の「今すぐ同期」。連絡先PIIや申込はクラウド側を温存(matchesと公開情報のみ反映)。
6. **バックアップ**: 進行中は7分毎に `data/backups/` へ自動スナップショット。終了後、本部のDB(`data/`)を保全。

### 事前準備チェック
- [ ] 本部PCに Node.js LTS と本アプリ(node_modules同梱)を入れて**オフラインで起動できる**ことを事前確認。
- [ ] テザリング/モバイルルータで**ローカル網**を作り、別端末から `http://本部IP:3000/admin` が開けることを確認。
- [ ] 本部PCの**スリープ/画面オフを無効**化(電源接続、`caffeinate`(Mac)等)。サーバが落ちると全端末が止まる。
- [ ] 申込データを事前にクラウド→本部へ取込(組合せ表Excel/スナップショット)しておく。
- [ ] (公開する場合)`SYNC_CLOUD_URL`/`SYNC_KEY` を設定し、事務所のネットで一度同期テスト。

### トラブル時
- **他端末が繋がらない**: 同じローカル網(本部テザリング/ルータ)に繋いでいるか確認。URLのIP/ポートを再確認。
- **本部PCが固まった**: 再起動 → `start.command` 再実行 → `data/` のDBはそのまま、最悪は `data/backups/` の最新を `deploy/restore.sh` で復元。
- **クラウド同期が進まない**: インターネット側が落ちているだけ。ローカル運用は継続し、戻れば自動同期される(「今すぐ同期」でも可)。

## 3. 初期セットアップ手順

### ステップ1: GAS スプレッドシート設定 (40分)

1. Google ドライブで新規スプレッドシート作成
   - 名前: 「KTTA 卓球大会 2026」など
2. メニュー: 拡張機能 → Apps Script
3. `gas/entry_form_handler.gs` の内容を全選択コピー → 貼付け
4. プロジェクト設定 (左メニュー歯車) → スクリプトプロパティ
   ```
   ADMIN_EMAIL    = kttatakkyu@example.com
   ASSOCIATION_NAME = 釧路卓球協会
   ISSUER_NAME    = 釧路卓球協会 会長 山本 満
   PRICE_TEAM_M   = 1000
   PRICE_TEAM_F   = 1000
   PRICE_DBL_M    = 1000
   PRICE_DBL_F    = 1000
   PRICE_MIX_M    = 500
   PRICE_MIX_F    = 500
   PRICE_BENTO    = 800
   PRICE_PARTY    = 3500
   ```
5. デプロイ → 新しいデプロイ → 種類: ウェブアプリ
   - アクセス: **全員 (匿名含む)**
   - 実行ユーザー: **自分**
6. デプロイ URL をコピー (例: `https://script.google.com/macros/s/AKfycb.../exec`)

### ステップ2: KTTA Platform サーバー設定 (30分)

#### Oracle Cloud 編 (現行の本番構成)
ORACLE_CLOUD_DEPLOY.md の手順どおり。要点:
1. GitHub にリポジトリを push (GitHub Actions の Secrets に ORACLE_HOST/ORACLE_SSH_KEY/ORACLE_USER)
2. サーバーで `deploy/install.sh` を実行 (Node/nginx/systemd を構成)
3. `/etc/ktta.env` に `NODE_ENV=production` `ADMIN_KEY=ランダムな20文字` 等を設定
4. main へ push すると自動デプロイ → URL を確認

#### VPS 編 (Vultr 1GB 例)
```bash
# Ubuntu 22.04 に SSH 接続後
curl -fsSL https://deb.nodesource.com/setup_20.x | sudo -E bash -
sudo apt install -y nodejs git nginx certbot python3-certbot-nginx
git clone <YOUR_REPO> /opt/ktta
cd /opt/ktta && npm install --omit=dev
# systemd ユニット作成
sudo cp deploy/ktta.service /etc/systemd/system/
sudo systemctl enable --now ktta
# Nginx + HTTPS
sudo cp deploy/nginx.conf /etc/nginx/sites-available/ktta
sudo ln -s /etc/nginx/sites-available/ktta /etc/nginx/sites-enabled/
sudo certbot --nginx -d kttatakkyu.example.com
```

### ステップ3: 管理画面 初期設定 (10分)

1. `https://yourdomain/admin/` を開く
2. 管理キー入力 (本番: `/etc/ktta.env` の ADMIN_KEY / VPS: settings 画面)
3. 設定アイコン (⚙) → 印鑑画像アップロード
4. 「本番URL設定」 → 公開ドメイン入力
5. **「登録団体マスタ（取込で団体を選手にしない）」** に、よく出る団体（クラブ・学校）名を登録しておく
   - ここに登録した名前は、組合せ表の取込で**選手名として取り込まれず所属（チーム名）扱い**になる。
     特にダブルスの相方欄に団体名が入る誤取込を防げる（最初に入れておくほど取込が安定する）。
6. 大会テンプレートから 1つ作成して動作確認

## 4. 大会ごとの運用フロー

### フェーズ1: 大会告知 (2ヶ月前 〜 申込開始)

1. **管理画面 → 大会一覧 → 「+ 新規大会」**
2. テンプレートから選択 (例: 「会長杯」)
3. 大会情報入力: 名前・日付・会場
4. 「保存」 → 大会ステータス = **予定**

### フェーズ2: 申込フォーム公開 (申込開始日)

5. **大会パネル → 「申込フォーム生成 / 埋込コード」**
6. 種目・料金を確認・調整 (カタログから追加 or テンプレ読込)
7. GAS Web App URL を貼付け
8. 締切日を設定
9. プラットフォーム = **Jimdo (推奨)** を選択
10. 「コードをコピー」 → Jimdo の HTML埋込ブロックに貼付け
11. 「申込受付 設定」 → ON にする → 種目チップ確認

**チェックポイント:**
- ✅ Jimdo で実際にフォームが表示されるか
- ✅ テスト送信して GAS スプレッドシートに記録されるか
- ✅ 自動返信メールが届くか

### フェーズ3: 申込受付期間中 (受付中 〜 締切)

申込状況を定期確認:

12. **管理画面 → 大会パネル → 「申込集計を取得」**
    - 申込団体数・申込人数・参加料合計が表示
13. スプレッドシート側を直接確認: 集計用・選手名簿シートで全体把握
14. 不明点や訂正があれば、申込者にメール返信

### フェーズ4: 締切後・組合せ表作成 (1週間前〜前日)

組合せ表の作り方は **2通り**。どちらも結果は同じ「ブラケット」になり、当日進行・観戦ビュー・対戦票で使える。

**(あ) システムで抽選ドロー → Excelで微調整(推奨・for_mac代替)**
15a. 申込管理で出場者を承認(confirmed)し、**シードを付ける選手にシード番号(1,2,3…)を指定**。
    - シード候補に迷ったら **「シード候補」**(Eloレーティング＋過去成績から根拠付きで提案。自動確定せず人手で採否)を参考にできる。
    - **スーパーシード**(1回戦を免除し N回戦から登場)を使う大会(なごやか亭杯・VICTAS杯 等)では、種目タブの **「登場R」** 列で各選手の **登場回戦** を指定し、**「🔍 登場回戦の配置をプレビュー」** で配置を確認してから抽選する。紙の組合せ表は「罫線が大きい＝後の回戦から登場」が目印。
16a. 進行管理の **トーナメント表編集 → 「🎲 抽選ドロー」** を押す(ウィザードが開く)。
    - **事前検査**が自動表示(承認漏れ・シード重複などを抽選前に警告=後で全やり直しを防ぐ)。
    - **① 分散方法**を選ぶ: **同一所属(クラブ)を散らす**(既定・推奨) / 地区 / 完全ランダム。
    - **② 「抽選してみる(確定しない)」** で組合せをプレビュー(まだDBに反映されない)。気に入らなければ **「引き直す」**。同じ並びを再現したいときは表示の **記録ID(draw_seed)** を入力。
    - **③ 実施者名**を入力(記録用・必須)。
    - **④ 「この並びで確定」**。シードは標準位置に固定、BYE(不戦勝)は上位シードに付く。確定は `draw_log` に記録(誰が・いつ・どの種で・何回引き直したか)。間違えたら **「直前の抽選を取消」** で元に戻せる。
17a. **「📋 表をExcel出力」**(両山)で Excel を取得 → 細部を手修正(氏名訂正・入替) → **「📥 Excel取込で反映」** で取り込み、組合せを正本化(選手データ・結果は保持・位置だけ更新)。画面上でも 1回戦のドラッグ入替・クリック編集が可能。

> **当日サーバ/ネットが落ちたら**: `standalone/start.command`(Mac) / `start.bat`(Win)でローカル単機起動すれば、ネット不通でも「名簿→抽選→Excel出力」まで完結できる(for_mac.xls に戻らずに済む)。

**(い) 外部で作った組合せ表を取込む(従来)**
15b. Excel/PDF で組合せ表を作成(「種目別 選手リスト」でシード順を決定) → **「組合せ表 Excel アップロード」** → **取込プレビュー（ファクトチェック）** → 取込実行。
    - プレビューは**種目別の出場者一覧**と、読み取り違いの可能性を **⚠ 要確認** として表示する(取込前に気づける):
      **組番号の欠番**(取りこぼし/欠場の可能性)・**氏名重複**(同一人物の二重 or 同姓同名)・**相方欠落の可能性**(ダブルス)。
    - 表の並び順そのまま(as_drawn)で取り込まれる。読み取り違いがあれば**取込後にトーナメント管理で修正**できる(出場者・結果は保持・位置だけ更新)。

16. 管理画面で大会ステータスを **「準備中」** に変更
18. 出場選手と試合 (1回戦〜決勝) が自動生成される
19. **「観戦ビュー / リアルタイム共有」** から QR コード取得
20. 大会案内 (印刷物・メール) に観戦ビュー URL/QR を記載

### フェーズ5: 大会当日 (運営)

#### 開会前 (1時間前〜)
- **受付係**: スプレッドシートから「全団体の領収書を一括発行」 → 印刷 → 当日受付で配布
- **当日キャンセル・追加対応**:
  - 個別領収書: スプレッドシートメニュー「個別領収書を発行」
  - 試合追加: 管理画面「+ 試合を追加」

#### 開会後・進行中
- **進行係**: 管理画面 → 大会パネル → 「進行管理を開く」
  - 「呼べる対戦」リストから台に割当
  - 試合終了後、結果入力 (5/3セット切替対応)
  - 敗者は自動で次の試合の審判候補へ
- **観戦案内係**: 参加者に観戦ビュー URL を案内
  - 自分の試合がどの台で行われているか検索可能
  - まだ呼べない試合には **「⏳ 待機: …」** と理由が表示される(例: 「○○ で試合中」「○○ で勝ち上がり中」「○○ で審判担当中」)。
    上位種目が片付くまで待つ仕組みなので、選手・保護者にも「なぜまだ呼ばれないか」が伝わる。
- 大会ステータスを **「進行中」** に切り替え (観戦ビューで赤バッジ表示)

#### 結果記録
- 各試合: 「結果入力」ボタン → 勝者選択 → セットカウント入力
- 任意で詳細点数入力可能 (例: 11-9, 8-11, 12-10, 9-11, 11-7)
- 入力後、自動で次ラウンドに勝者を送る
- **団体戦**: 種目作成時に対戦形式 `tie_format`(例 `S,S,D,S,S` = シングルス/ダブルスの並び、空欄ならチームスコア直接入力)を設定すると、進行管理に**個別試合(1複4単など)の勝敗を入れる専用モーダル**が出て、勝った試合数(団体スコア)で勝者チームを次戦へ送る。個別試合の台割・オーダーは紙運用(アプリ管理しない)。

### フェーズ6: 大会終了後

- 大会ステータスを **「終了」** に変更
- **集計表をダウンロード** (Excel) → 協会保管
- **領収書一覧** をスプレッドシートから印刷保管 (会計用)
- 必要に応じて **大会結果 JSON エクスポート** → アーカイブ

## 4.7 選手マージ（重複統合）運用

### いつ使うか
- 同一人物が複数IDで登録されている（例: 名前の表記ゆれで別々に作成）
- 「重複候補」ページで候補が表示された場合

### 手順（管理画面 → 🔒 システム管理（オーナー） → 選手重複の統合）
1. 重複候補グループを確認する（同一人物かを必ず判断してから操作する）
2. **残す選手（survivor）= 先に登録された方（古いID）** が既定で選ばれる → 問題なければそのまま確定
3. 確定後: 旧IDの全成績・所属リンクが survivor へ付け替わる。旧IDはシステムに残り、旧URLで閲覧しても survivor のページに自動リダイレクトされる

### 取り消し（誤統合した場合）
1. 同じオーナーパネル内の「選手マージ履歴 / 取り消し」から対象の統合を探す
2. **新しい順（LIFO）にのみ取り消せる** — 古いマージを先に取り消したい場合は、それより新しいマージを先に取り消すこと
3. 「取り消す」を押すと付け替えた行のみ元に戻り、survivor に新たに蓄積された成績は移動しない（設計上正しい動作）

### 注意事項
- 旧実装（2026年6月以前）で物理削除された dup は遡及して取り消せない
- 誤統合を防ぐため、必ず同一人物であることを確認してから操作すること
- 統合後に survivor へ蓄積した成績は、取り消し時に dup へ移らない（意図した挙動）

## 5. 役割分担 (推奨)

| 役割 | 担当者数 | 主な作業 | 必要なもの |
|---|---|---|---|
| 大会本部 | 1名 | 全体統括・進行管理 | PC + KTTA管理画面 |
| 受付 | 1〜2名 | 領収書配布・出席確認 | 印刷した領収書 |
| 進行係 | 1〜2名 | 台割当・呼出 | タブレット or PC |
| 試合記録 | 各台1名 | 結果入力 | スマホ (各台担当) |
| 観戦案内 | 1名 | 観戦ビュー URL 案内 | 印刷物 + QR コード |

### スマホ運用について
- 観戦ビュー (`/viewer/live/`): 参加者・保護者全員
- 進行管理 (`/admin`): 運営スタッフのみ (管理キー必須)
- iPad / iPhone 推奨。Android も対応

## 6. バックアップ運用 (重要)

### 自動バックアップ (毎日)
```bash
# crontab -e で設定
0 2 * * * /opt/ktta/deploy/backup.sh
```
バックアップ先: **`/var/data/backups/ktta-YYYYMMDD-HHMMSS.db.gz`**(本番DBは `/var/data/tournament.db`。
backup.sh/restore.sh は DB_PATH→/var/data を自動判定する)
- 30日分自動ローテーション
- 大会前日は手動で 1回バックアップ推奨: `sudo /opt/ktta/deploy/backup.sh`

### ⚠ オフサイト退避 (必須級・全損対策)
既定では全バックアップが本機 `/var/data` に同居し、VM/ディスク喪失・誤削除で**全損**する。別ロケーションへ
1コピー退避すること。`/etc/ktta.env` に転送設定を入れ、backup.sh が毎回退避する:
```bash
# 例: OCI Object Storage(別アカウント/別リージョン推奨)
OFFSITE_CMD='oci os object put -bn ktta-backups --force --file'
# 例: S3互換   OFFSITE_CMD='aws s3 cp'   OFFSITE_DEST='s3://ktta-backups/'
# 例: rclone   OFFSITE_CMD='rclone copy' OFFSITE_DEST='myremote:ktta-backups'
```
※ cron 実行は env を継承しないため、`0 2 * * * . /etc/ktta.env; /opt/ktta/deploy/backup.sh` のように
env を source してから実行する。

#### お名前ドットコム(レンタルサーバー SSH/SFTP)へ毎晩退避する手順
Oracle無料枠VMの回収に備え、別事業者(お名前ドットコム)へ DB を毎晩ミラーする。専用スクリプト
`deploy/offsite-sync.sh`(ローカルの30日ローテ済バックアップを rsync ミラー。空ソース時は中止する安全ガード付き)を使う。

```bash
# 1) Oracle 側で ktta 用の SSH 鍵を作成(パスフレーズ無し=cron用)
sudo -u ktta ssh-keygen -t ed25519 -f /home/ktta/.ssh/onamae_rsync -N "" -C "ktta-offsite"
sudo -u ktta cat /home/ktta/.ssh/onamae_rsync.pub   # ← この公開鍵を控える

# 2) お名前ドットコム側に公開鍵を登録(コントロールパネルのSSH設定 or 下記)
#    ssh <user>@<onamae-host> 後、~/.ssh/authorized_keys に上の .pub を追記。退避先フォルダも作る:
#    mkdir -p ~/ktta-backups && chmod 700 ~/.ssh

# 3) Oracle 側で疎通確認(初回はホスト鍵を accept)
sudo -u ktta ssh -i /home/ktta/.ssh/onamae_rsync -p <PORT> <user>@<onamae-host> 'echo OK; ls ~/ktta-backups'

# 4) /etc/ktta.env に退避設定を追記
OFFSITE_DEST=<user>@<onamae-host>:~/ktta-backups/
OFFSITE_SSH_KEY=/home/ktta/.ssh/onamae_rsync
OFFSITE_SSH_PORT=<PORT>          # お名前RSは 2222 等の場合あり。標準SSHなら22

# 5) 毎晩3時に「ローカルバックアップ → お名前ドットコムへミラー」する cron を登録(ktta ユーザーの crontab)
#    crontab -u ktta -e
0 3 * * * . /etc/ktta.env; /opt/ktta/deploy/backup.sh && /opt/ktta/deploy/offsite-sync.sh >> /var/data/backups/offsite.log 2>&1

# 6) 手動で1回流して確認
sudo -u ktta bash -c '. /etc/ktta.env; /opt/ktta/deploy/backup.sh && /opt/ktta/deploy/offsite-sync.sh'
```
- rsync があれば `--delete` で遠隔も同じ30日窓に自動ローテ。rsync 不在時は scp で全件転送(遠隔ローテは手動)。
- **災害復旧(VM回収時)**: 新サーバを建てて配置(git)後、お名前ドットコムから最新 `ktta-*.db.gz` を取得し、
  管理画面「🔒 システム管理（オーナー）」→ バックアップ/復元 → **「ファイルから復元（アップロード）」** で
  その `.db.gz` を取り込めば復旧する(ローカルにスナップショットが無くてもよい)。CLI なら restore.sh にパス指定も可。

### 復元 (restore.sh を使う。パスは自動解決)
```bash
sudo /opt/ktta/deploy/restore.sh --list                 # 一覧(/var/data/backups を参照)
sudo /opt/ktta/deploy/restore.sh 20260711-020001        # タイムスタンプ指定で復元
sudo /opt/ktta/deploy/restore.sh /path/to/ktta-xxx.db.gz # 別ロケーションから取得したファイルを直接指定
```
restore.sh は復元前に現DBを `before-restore-*.db` へ自動退避し、サービス停止→復元→再起動まで行う。
**重要**: 復旧手順は障害発生時に初めて使うと失敗しがち。**非番時に1回ドライ実行**して `--list` が
`/var/data/backups` を指し、復元先が `/var/data/tournament.db` になることを確認しておくこと。

## 7. トラブルシューティング

### Jimdo でフォームが表示されない
1. 管理画面 → 「申込フォーム生成 / 埋込コード」 → 右下 **「本番URL設定」** で https:// から始まる URL を設定
2. **Jimdo モード** で生成 (iframe 非使用なので HTTPS 問題回避)
3. それでも表示されない場合: ブラウザの DevTools (F12) でエラー確認

### スプレッドシートに申込が届かない
1. GAS Web App URL が正しく貼付けられているか確認
2. GAS のデプロイが **「全員」アクセス** になっているか
3. GAS のログ: スプレッドシート → 拡張機能 → Apps Script → 実行 → ログ

### 進行管理の試合が呼べない
- 「優先順位ロック」発動中: 上位種目 (団体・ミックス・ダブルス) で生存中の選手はシングルスでは呼べない
- 当該選手の上位種目が完了するまで待機 (緑色「呼出可能」表示まで)
- **理由の確認**: 観戦ビュー/進行管理で当該試合に **「⏳ 待機: ○○ で試合中／勝ち上がり中／審判担当中」** と理由が出る。審判で拘束されている場合は呼出行の **「審判を解放」** で解任すれば呼べる。どうしても先に進めたい場合のみ `force` で強制呼出可。
- **呼出は大会ステータスが「進行中」のときだけ可能**(予定/準備中では呼べない)。

### サーバーが応答しない
```bash
sudo systemctl status ktta       # 状態確認
sudo systemctl restart ktta      # 再起動
sudo journalctl -u ktta -n 100   # ログ確認
```

### データ破損 (滅多に起こらない)
- 即座にバックアップから復元
- WAL モードで運用しているため、突然の停電でもデータ破損は稀
